# Address PR Review

GitHub Pull Request の未解決レビューコメントを起点に、要約, 対応選択, 計画立案, 並列修正, コミット/PR更新までを行うスキル。

## このスキルが想定する状況

ユーザーが自分の作業ブランチで作業しており、その PR にレビューコメントが付いている。コメントは複数あり、対応すべきもの・無視すべきもの・自明な微修正が混在する。ユーザーは個別に対応指示を出すよりも、まとめて捌きたい。
このスキルは「ユーザーは全部のコメントに目を通している暇がない」「でも自動で全部直すのは怖い」という間の温度感を埋めるためにある。だから要約と選択ステップを省略してはいけない。逆に明らかな typo 修正でいちいち確認を求めるのも邪魔なので、後述の trivial 判定でフィルタする。


## 1. 未解決コメントの取得

このスキルに同梱されている `scripts/fetch_unresolved_comments.sh` を実行する。
ユーザーが PR 番号を指定している場合はそれを第1引数に、そうでなければ無引数で実行する。

出力 JSON 例:

```json
{
  "pr":      { "number": 1234, "url": "...", "title": "...", "owner": "...", "repo": "..." },
  "threads": [ { "isOutdated": false, "path": "src/a.ts", "line": 42, "url": "...", "author": "alice", "body": "...", "diffHunk": "...", "commentId": "..." } ],
  "reviews": [ { "author": "bob", "state": "CHANGES_REQUESTED", "body": "...", "url": "...", "submittedAt": "..." } ]
}
```

各 thread はスレッドの先頭コメントのみを返す（リプライは URL から辿る）。`isOutdated: true` な
スレッドは残してあるので、対応要否はユーザー判断に委ねる（リスト表示時にその旨を注記）。


## 2. 要約

取得したコメント群と review body を日本語で200字以内に要約してユーザーに報告する。


## 3. 対応選択

未解決コメントをリスト化し、AskUserQuestion の `multiSelect: true` で「対応するもの」を選ばせる。デフォルトは全部未選択とする。
各 option の `label` は短く（指摘の主旨1行）、`description` にコメント本文の要点とファイル:行を入れる。

以下のいずれかに該当するコメントは選択を求めず、自動的に「対応する」として後段の plan に含める。
判定に迷ったら trivial 扱いせず通常リストに入れる。ユーザーの意図しない変更が混ざることを防ぐため、保守的に判断する。

- typo / 誤字脱字 の指摘
- lint / format / インポート順 などスタイル系
- コメント本文に `nit:` `minor:` `super minor:` などの prefix が付いているもの
- 修正範囲が明らかに 1〜2 行で完結し、ロジックに影響しないもの


## 4. Plan モードで修正案を提示

Plan Mode を使い、選ばれたコメントごとに「どのファイルをどう変えるか」を提示する。
具体的に実行可能で、解釈の余地がない修正案を作成する。

並び順:
1. ユーザーが明示的に選んだ修正（重要度の高そうな順）
2. Trivial として自動採用した修正（まとめてセクション化）


## 5. 並列修正の実行

Plan が承認されたら、項目ごとに Agent ツールでサブエージェントを起こす。原則として独立した項目は同時に投げる（並列化）。

### モデル選択

- **sonnet** (`subagent_type: claude`, `model: sonnet`): typo 修正、リネーム、スタイル修正、1-2行の置換、明確な行追加など機械的なタスク。
- **opus** (`subagent_type: claude`, `model: opus`): ロジック変更、複数ファイルにまたがる修正、設計判断を含むリファクタ、テスト追加でケース設計が必要なもの。

判定に迷う場合、Opus を選択する。

### 並列化の制約

同じファイルを編集する項目は直列にする（書き込み競合を避けるため）。それ以外は並列で投げる。各サブエージェントには：

- 該当コメントの URL と本文
- 対象ファイル:行
- 修正方針（plan からの抜粋）
- 「テストや lint は自分で走らせて確認すること」「変更箇所以外には触れないこと」

を明示的に渡す。

### 完了後の確認

全サブエージェント完了後、`git diff` を眺めて意図しない変更や衝突がないか確認する。重大な逸脱があればユーザーに報告。


## 6. コミットと PR 更新

AskUserQuestion で次を二択（または三択）で確認：

- コミットして PR を更新
- コミット後に履歴を整理してから PR を更新: 修正箇所を対応する既存のコミットに取り入れる。

コミット後に push まで行う。PR へのコメントへの返信や resolve は行わない。

### ユーザーが追加修正を指示してきた場合

コミットを保留し、追加修正を実装した上で、ステップ7を最初から実行する。

## エラーハンドリング

- `gh` 未インストール / 未ログイン → ユーザーに案内
- PR が見つからない → ブランチ名と検索条件を報告
- 未解決コメントがゼロ → 「対応すべき未解決コメントはありません」と報告して終了
- サブエージェントが失敗した項目 → リストアップしてユーザーに判断を仰ぐ（自動リトライしない）

## やらないこと

- レビュアーへの返信投稿（明示指示なき限り）
- スレッドの自動 Resolve
- main / master への直接 push
- ユーザーが選択しなかったコメントへの「ついで対応」
